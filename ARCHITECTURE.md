# Архитектура сайта WYRD

## Статус документа

- Формат: рабочий редактируемый документ
- Назначение: базовый источник истины по текущей архитектуре сайта
- Актуальность: зафиксировано по состоянию текущего кода в проекте
- Правило обновления: при изменении структуры, экранов, состояния или пользовательских сценариев сначала обновлять этот файл, затем код либо вместе с кодом
- Связанный индекс документации: `docs/README.md`

---

## 1. О продукте

`WYRD` — это одностраничный мистический сайт-оракул с офлайн-поддержкой на базе `HTML/CSS/JavaScript` без сборщика и без фреймворков.

Ключевая продуктовая идея:

- пользователь входит через cover-screen
- открывает бесплатную карту дня
- может запускать дополнительные сценарии: ещё карта, углубление значения, расклады на 3 и 5 карт
- просматривает профиль и локальную историю
- состояние хранится локально в браузере

## 2. Технологический стек

- `index.html` как единая HTML-точка входа
- `assets/css/styles.css` как единый файл стилей
- `ES Modules` в браузере
- `localStorage` для клиентского состояния
- `Web Audio API` для фонового лесного звука и шуршания карт
- `Service Worker + manifest` для PWA-режима

Архитектурный стиль:

- статический SPA без роутера
- модульная клиентская архитектура
- ручное управление состоянием и DOM

## 4. Архитектурные слои

Сайт уже разделён на несколько понятных слоёв.

### 4.1. Presentation Layer

Файлы:

- `index.html`
- `assets/css/styles.css`
- `assets/js/ui/render.js`

Ответственность:

- статическая разметка экранов
- визуальные состояния интерфейса
- показ/скрытие секций
- отрисовка текущей карты, расклада, истории, paywall и профиля

### 4.2. Interaction Layer

Файлы:

- `assets/js/ui/actions.js`

Ответственность:

- слушать клики по `data-action`
- маршрутизировать пользовательские действия
- переключать overlay-состояния
- запускать сценарии чтения и раскладов

### 4.3. Domain Layer

Файлы:

- `assets/js/cards/reading.js`
- `assets/js/cards/oracle-config.js`

Ответственность:

- создать чтение одной карты
- создать расклад на несколько карт
- определить тексты локального paywall-preview
- хранить настраиваемый probability-profile оракула в отдельном конфиге

### 4.4. State Layer

Файлы:

- `assets/js/state/storage.js`

Ответственность:

- загрузка состояния из `localStorage`
- нормализация состояния
- обновление состояния через методы store
- синхронизация дневной бесплатной карты по границе суток

### 4.5. Infrastructure Layer

Файлы:

- `assets/js/audio.js`
- `assets/js/pwa.js`
- `sw.js`
- `manifest.webmanifest`

Ответственность:

- звук
- офлайн-кэширование
- регистрация service worker
- PWA-обвязка

### 4.6. Content/Data Layer

Файлы:

- `assets/js/data/cards.js`

Ответственность:

- единый источник истины по картам
- хранение путей к изображениям
- хранение текстового контента карты

## 5. Главные модули и их роли

## `assets/js/main.js`

Роль:

- собирает приложение из модулей
- инициализирует store, ui state, renderer, audio
- навешивает глобальный click handler
- запускает первый render
- регистрирует service worker

Это тонкий composition root. Здесь не должна жить предметная логика.

## `assets/js/data/cards.js`

Роль:

- канонический набор карт
- хранит только сами карты, тексты и изображения
- каждая карта содержит:
  - `id`
  - `name`
  - `subtitle`
  - `keyword`
  - `state`
  - `layer`
  - `message`
  - `shadow`
  - `image`

Ограничение:

- сейчас данные жёстко зашиты в коде
- в текущей реализации в игре подключены карты `1–48`
- часть новых карт пока использует пустой visual-slot до загрузки финальной иллюстрации
- расширенный список персонажей для следующей итерации продукта ведётся в этом документе как master-roster
- бесплатная карта дня выбирается только из слоя `present`
- расклад `3` карт строится как `past / present / future`
- расклад `5` карт строится как `present / past / present / random / future`
- расклад `3` карт раскрывается поэтапно в порядке `center -> past -> future`
- расклад `5` карт рендерится как ритуальная композиция по позициям, а не как простая равномерная сетка
- выбор карты больше не равновероятный: используется weight-модель по `layer + tone + anti-duplicate state + links`
- веса, multipliers и схемы раскладов вынесены в `assets/js/cards/oracle-config.js`
- `tone/state/links` вынесены из колоды в отдельный `assets/js/cards/card-meta.js`
- `layer` вынесен из колоды в отдельный `assets/js/cards/layer-map.js`

## 6. HTML-структура и экранная модель

Сайт фактически работает как один экран с переключаемыми секциями.

### Базовые зоны интерфейса

- `#cover` — стартовый экран
- `#main` — основная оболочка приложения
- `#deck-wrap` — колода и основной CTA на вытягивание карты
- `#result` — показ одной текущей карты
- `#spread-result` — расклад на несколько карт
- `#paywall` — overlay-секция локального paywall-preview
- `#profile` — overlay-секция профиля и истории

### Модель экранов

Не используется отдельный роутер. Вместо этого состояние UI строится из двух координат:

- `uiState.overlay`
- `contentPanel`, вычисляемый из store state

Значения `overlay`:

- `none`
- `paywall`
- `profile`

Значения `contentPanel`:

- `deck`
- `result`
- `spread`

Итог:

- основной контент и overlay логически разделены
- routing организован как state-driven visibility

## 7. Модель состояния

Источник: `assets/js/state/storage.js`

```js
{
  profileName: "Странник",
  soundEnabled: true,
  ambienceVolume: 0.58,
  dailyFreeUsedAt: null,
  history: [],
  currentReading: null,
  lastSpread: []
}
```

### Значение полей

- `profileName` — имя пользователя в профиле
- `soundEnabled` — глобальный флаг аудио
- `ambienceVolume` — уровень окружения
- `dailyFreeUsedAt` — ISO-дата использования бесплатной карты
- `history` — история чтений одиночных карт
- `currentReading` — текущая активная одиночная карта
- `lastSpread` — текущий активный расклад

### Инварианты состояния

- если есть `lastSpread`, то `currentReading = null`
- если есть `currentReading`, то `lastSpread = []`
- при сохранении чтения сбрасывается предыдущий расклад
- при сохранении расклада сбрасывается текущее одиночное чтение

### Ограничения текущей модели

- история хранит только одиночные чтения
- нет отдельной модели пользователя
- нет версионирования схемы состояния кроме ключа `wyrd-local-state-v2`
- нет слоя миграций для будущих изменений состояния

## 9. Пользовательские сценарии

### 9.1. Вход в приложение

1. Пользователь нажимает `Войти в лес`
2. `uiState.entered = true`
3. Выполняется render
4. При включённом звуке инициируется аудио-контекст

### 9.2. Бесплатная карта дня

1. Пользователь нажимает на колоду
2. Проверяется `store.hasFreeDraw()`
3. Если карта дня ещё не использована:
   - выставляется `dailyFreeUsedAt`
   - создаётся новое `reading`
   - чтение попадает в `currentReading`
   - чтение добавляется в `history`
   - UI скроллится к результату

### 9.3. Дополнительные платные сценарии

1. Пользователь нажимает:
   - ещё карта
   - углубить значение
   - расклад на 3 карты
   - расклад на 5 карт
2. Открывается локальный preview paywall
3. По `confirm-paywall` выполняется действие

Важно:

- это не настоящий биллинг, а продуктовый placeholder

### 9.4. Углубление значения

1. Пользователь подтверждает paywall
2. Для текущего чтения `depthUnlocked = true`
3. В UI показывается дополнительный текст `DEEP_READING_TEXT`

### 9.5. Расклад

1. Пользователь выбирает расклад на 3 или 5 карт
2. Генерируется случайный набор карт
3. Сохраняется в `lastSpread`
4. UI переключается в `spread`

### 9.6. Профиль и история

1. Пользователь открывает профиль
2. Появляется overlay профиля
3. Внутри выводится статус бесплатной карты и история чтений

### 9.7. Сброс локальных данных

1. Пользователь нажимает `Очистить локальные данные`
2. Аудио останавливается
3. Store сбрасывается в default state
4. Cover-screen показывается заново

## 11. Правила текущей архитектуры

Эти правила стоит сохранить и использовать дальше.

### 11.1. Что где должно жить

- новые данные карт и контент карт: `assets/js/data/cards.js`

- не складывать бизнес-логику обратно в `main.js`

- не смешивать изменение state и низкоуровневый DOM-рендер в одном месте без причины
- не добавлять новые пользовательские сценарии напрямую в HTML без маршрутизации через `data-action`

## 12. Слабые места и технический долг

### 12.1. Неподключённый модуль `ritual.js`

Проблема:

- есть отдельный контроллер ритуала, но он не используется

Следствие:

- архитектура намекает на сценарий ожидания/отсчёта, но в реальном UX его нет

### 12.2. Неподключённый аудиофайл

Проблема:

- `assets/audio/WYRD.m4a` лежит в проекте, но код использует синтезированный звук

Следствие:

- актив проекта не встроен в продуктовый поток

### 12.3. Нет реального платёжного слоя

Проблема:

- paywall только визуальный и логический placeholder

Следствие:

- монетизация не отделена как подсистема

### 12.4. Нет тестового слоя

Проблема:

- отсутствуют unit/integration/e2e тесты

Следствие:

- при изменении логики состояния и UI легко сломать сценарии

### 12.5. Ручной DOM-рендер

Проблема:

- прямой DOM-подход прост, но по мере роста сценариев усложнит поддержку

Следствие:

- выше риск разрастания `render.js` и `actions.js`

### 12.6. Service Worker поддерживается вручную

Проблема:

- новые файлы нужно вручную добавлять в `CORE_ASSETS`

Следствие:

- офлайн-режим может отставать от фактической структуры проекта

## 13. Рекомендованная целевая архитектура развития

Если сайт будет расти, лучше развивать его в этих направлениях.

### Этап 1. Укрепить текущую архитектуру без смены стека

- вынести продуктовые сценарии в отдельный слой `features/` или расширить `cards/`
- подключить `ritual.js` как реальный промежуточный state между нажатием и результатом
- добавить отдельную структуру для `spread` с `id`, `type`, `createdAt`, `cards`
- сохранять историю не только одиночных чтений, но и раскладов
- выделить конфиг приложения: лимиты, тексты CTA, фичи, paywall-режим

### Этап 2. Добавить управляемость

- завести папку `docs/` для продуктовой и технической документации
- добавить `CHANGELOG_ARCHITECTURE.md` или блок change log внутри этого файла
- описать соглашения по именованию action, state-полей и секций UI
- ввести минимальные smoke-тесты на основные сценарии

### Этап 3. Если проект вырастет

- рассмотреть переход на лёгкий сборщик (`Vite`) без обязательной смены UI-подхода
- разделить page shell, feature-модули и infra ещё жёстче
- при сильном росте интерактивности рассмотреть state machine или компонентный UI-подход

- дата
