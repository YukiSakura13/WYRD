# ADR: Architectural Decision Record

## Как использовать

Этот файл хранит важные архитектурные и продуктово-технические решения.

Формат записи:

- ID
- дата
- статус
- контекст
- решение
- последствия

Статусы:

- `accepted`
- `proposed`
- `deprecated`
- `superseded`

---

## ADR-001

- Дата: 2026-04-05
- Статус: `accepted`
- Тема: Базовый стек проекта

### Контекст

Проект уже реализован как лёгкий статический сайт/PWA без сборщика и без фреймворка.

### Решение

Сохраняем текущий стек как базовый:

- `HTML`
- `CSS`
- `Vanilla JS` через ES modules
- `localStorage`
- `PWA`

### Последствия

- проще поддерживать текущую кодовую базу
- быстрее делать небольшие продуктовые изменения
- нужно строже следить за дисциплиной модулей и документации

## ADR-002

- Дата: 2026-04-05
- Статус: `accepted`
- Тема: Документация как часть разработки

### Контекст

Нужно выстроить более плотную совместную работу с Codex, чтобы изменения опирались не только на код, но и на общий проектный контекст.

### Решение

Вводим обязательный набор опорных документов:

- `ARCHITECTURE.md`
- `docs/PROJECT_BRIEF.md`
- `docs/ROADMAP.md`
- `docs/FEATURES.md`
- `docs/CONTENT_MODEL.md`
- `docs/ADR.md`
- `docs/WORKFLOW_WITH_CODEX.md`

### Последствия

- у проекта появляется единый описанный контекст
- проще ставить задачи и отслеживать изменения
- документацию нужно обновлять регулярно, иначе она потеряет ценность

## ADR-003

- Дата: 2026-04-05
- Статус: `accepted`
- Тема: `ARCHITECTURE.md` как главный источник истины по структуре

### Контекст

Нужно, чтобы любые будущие изменения структуры сайта можно было быстро сверять с понятным документом.

### Решение

Главным архитектурным источником истины по текущей структуре считается:

- [../ARCHITECTURE.md](/Users/marinamart/Desktop/Oracle_dev/ARCHITECTURE.md)

### Последствия

- рефакторинги и новые сценарии проще обсуждать до кода
- архитектурные изменения должны сопровождаться обновлением документа
