# WYRD UI Rules

## Зачем нужен этот документ

Этот документ фиксирует правила интерфейса, motion и UX-полировки для `WYRD`.
Он нужен, чтобы проект не расползался визуально и чтобы повторяющиеся решения принимались по одной системе, а не заново каждый раз.

Он опирается на:
- текущую архитектуру `WYRD`
- accessibility-first подход проекта
- практики design engineering из внешней skill documentation

---

## 1. Базовый принцип

`WYRD` должен ощущаться как:
- ясный
- собранный
- ритуальный
- спокойный
- отзывчивый

Красота важна, но не ценой читаемости, скорости и понятности.

---

## 2. Что важнее всего

При любом UI-решении приоритет такой:

1. accessibility
2. readability
3. responsive behavior
4. semantic clarity
5. visual polish
6. delight

Если красивый эффект мешает чтению, навигации или мобильному UX, эффект убирается или упрощается.

---

## 3. Правила motion

### Анимируем только то, что помогает смыслу

Хорошие причины для анимации:
- показать появление нового слоя интерфейса
- поддержать ощущение раскрытия карты
- дать тактильный отклик на нажатие
- смягчить резкую смену состояния

Плохие причины:
- просто “чтобы было красиво”
- повторяющаяся декоративная анимация без функции
- движение, которое мешает читать расклад

### Частые действия должны быть быстрыми

Чем чаще пользователь видит анимацию, тем легче и короче она должна быть.

Правило:
- частые действия: почти без анимации или очень коротко
- редкие действия: можно мягче и атмосфернее

### Стандартные easing и duration

Используем общие токены проекта:
- `--ease-out-strong`
- `--ease-in-out-strong`
- `--ease-standard`
- `--dur-fast`
- `--dur-medium`
- `--dur-slow`
- `--dur-ritual`

Базовые правила:
- для UI-входа: `ease-out`
- для спокойных переходов состояния: `ease-in-out`
- не использовать `ease-in` как основной easing для UI
- не использовать `transition: all`

### Press feedback обязателен для интерактивных элементов

Кнопки и tappable-элементы должны ощущаться живыми.

Допустимый паттерн:
- `transform: scale(0.97)` на `:active`

Это должно быть:
- коротко
- мягко
- без “прыжка”

### Hover только для pointer:fine

Hover-эффекты применяются только под:

```css
@media (hover: hover) and (pointer: fine)
```

На touch-устройствах hover не должен создавать ложные состояния.

### Reduced motion обязателен

Для пользователей с `prefers-reduced-motion: reduce`:
- убираем или резко упрощаем movement-based анимации
- оставляем лёгкие opacity/transparency transitions
- не заставляем пользователя следить за движением

## 4. Правила для раскладов

### Расклад и чтение — это два разных слоя

В `WYRD` расклад должен строиться в два слоя:

1. `spread map`
   Это форма расклада, позиции и карты как узор.

2. `reading details`
   Это подробное чтение выбранной карты.

Нельзя перегружать саму карту расклада большим количеством текста.

### Голос Оракула живёт отдельно

`Голос Оракула`:
- не дублирует текст отдельных карт
- не показывается под каждой картой
- является синтезом всего расклада

### Мобильный UX важнее декоративности

На мобильном:
- форма расклада должна сохраняться
- текст не должен ломать геометрию расклада
- `3-card` должен оставаться треугольником
- `5-card` должен оставаться крестом

Если полный контент не помещается в карту, он переносится в отдельную панель ниже.

---

## 5. Правила компонентов

### Хорошие дефолты важнее настроек

Лучше один очень хороший дефолт, чем много опций.

Это относится к:
- анимациям
- размерам
- spacing
- поведению раскладов
- `Голосу Оракула`

### Компонент должен быть цельным по тону

Motion, типографика, интервалы и визуальные детали должны ощущаться как одна система.

Для `WYRD` это значит:
- без резких “технических” анимаций
- без случайной декоративности
- без современного “гладкого SaaS”-ощущения, если оно ломает атмосферу

### Edge cases должны решаться тихо

Нужно заранее учитывать:
- мобильные ширины
- старый кэш service worker
- touch вместо hover
- длинные тексты
- нестабильные размеры карточек

Пользователь не должен замечать эти проблемы.

---

## 6. Что не делать

- не использовать `transition: all`
- не анимировать keyboard-like быстрые действия тяжёлыми reveal-эффектами
- не класть длинный текст внутрь самой сетки расклада
- не смешивать `spread map` и `reading details`
- не делать hover как основной механизм для touch-устройств
- не добавлять сложную motion-логику без явной пользы
- не вводить новый визуальный паттерн без проверки на согласованность с уже существующим тоном `WYRD`

---

## 7. Практическое правило для будущих задач

Если задача касается:
- интерфейса
- анимации
- раскладов
- чтения карт
- мобильного UX
- `Голоса Оракула`

то перед изменениями нужно сверяться с этим документом.

---

## 8. Мини-чеклист перед UI-правкой

Перед каждым UI-изменением нужно спросить:

1. Это улучшает читаемость?
2. Это улучшает мобильное поведение?
3. Это не ломает форму расклада?
4. Это не создаёт лишнюю анимацию?
5. Это согласуется с тоном `WYRD`?
6. Это будет нормально работать с `prefers-reduced-motion`?
7. Это действительно нужно, или это просто декоративное усложнение?

---

## 9. Статус документа

Это живой рабочий стандарт проекта.
Его можно дополнять, когда:
- меняется UI-модель
- появляется новая модель расклада
- уточняется логика `Голоса Оракула`
- меняются базовые motion-принципы
