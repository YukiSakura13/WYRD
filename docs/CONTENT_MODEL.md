# Content Model: WYRD

## Назначение

Этот документ описывает, какие сущности контента уже есть в проекте и как их лучше развивать дальше.

---

## Текущие контентные сущности

### Card

Сейчас карта хранится в `assets/js/data/cards.js` и содержит:

```js
{
  id,
  name,
  subtitle,
  keyword,
  state,
  layer,
  message,
  shadow,
  image
}
```

На уровне продукта уже существует расширенный master-roster колоды:

- `1–48` уже подключены в игре
- `23–41` подтверждены по `wyrd_cards.md`
- `42–48` уже описаны и готовы к следующему расширению
- master-roster теперь непрерывен в диапазоне `1–48`
- часть новых карт пока содержит пустой `image` и рендерится как пустой visual-slot
- `layer` использует значения `past | present | future`
- `state` хранит смысловое состояние карты
- `tone` использует значения `light | neutral | dark`
- `links` хранит массив смысловых состояний, которые стоит мягко усиливать после этой карты
- probability-правила оракула теперь вынесены в отдельный конфиг `assets/js/cards/oracle-config.js`
- смысловой meta-слой карт вынесен в `assets/js/cards/card-meta.js`

### Cover Content

На стартовом экране сейчас используются:

- заголовок
- подзаголовок
- краткое позиционирование
- CTA-кнопка
- cover image

Часть контента зашита в `index.html`, часть в `cards.js`.

### Paywall Offer

В `assets/js/cards/reading.js` хранится:

```js
{
  title,
  text,
  cta
}
```

по ключам сценариев:

- `extra-draw`
- `deep-reading`
- `spread-3`
- `spread-5`

### Deep Reading

Сейчас это один шаблонный текст в `assets/js/ui/render.js`, в который подставляется имя карты.

---

## Текущие ограничения контентной модели

- контент распределён между `HTML`, `JS data` и `render`
- индивидуального deep-content у карт нет
- нет отдельной модели для расклада как контентной сущности
- нет формального описания позиций расклада
- нет отдельного контентного слоя для CTA, labels и системных текстов

---

## Рекомендуемая целевая модель

### Card

В будущем стоит расширить до такого вида:

```js
{
  id,
  slug,
  name,
  subtitle,
  keyword,
  message,
  shadow,
  deepMessage,
  image,
  tags,
  archetype
}
```

### Spread Definition

Полезно описывать расклады отдельной сущностью:

```js
{
  id,
  title,
  count,
  positions: [
    { key, title, meaning }
  ]
}
```

### UI Copy

Полезно вынести в отдельную структуру:

- тексты кнопок
- заголовки секций
- labels
- paywall copy
- profile copy

---

## Рекомендуемый формат хранения на будущее

Когда контента станет больше, лучше перейти к одному из вариантов:

- `data/cards.json`
- `data/cards.js` с более формальной схемой
- `data/content/` с разделением на `cards`, `spreads`, `ui-copy`

Оптимальный практический путь сейчас:

- оставить JS-модули
- но разделить на отдельные файлы по типам контента

---

## Принципы редактирования контента

- контент не должен дублироваться в нескольких местах
- каждый тип контента должен иметь один канонический источник
- если меняется структура карты или расклада, сначала обновляется этот документ
